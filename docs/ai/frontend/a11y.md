# Accessibility (a11y)

User-visible copy is **internationalised** (`$i18n.*` keys generated from
`src/lib/i18n/<locale>.json`). Accessibility, however, is non-negotiable
regardless of locale — this page is the contract.

## Copy & strings

- **Use `$i18n.*` for new copy.** Add the key to every locale JSON, run
  `npm run i18n`, then reference it. See
  [`workflows/i18n.md`](./workflows/i18n.md).
- **Brand / token names are exempt.** `Pandame`, `Juno`, `Internet
Identity`, etc. don't need to be wrapped — they don't translate.
- **Internal logs / error messages** are not user-visible and don't need
  the same discipline. Keep them readable for engineers.
- **No magic concatenation.** Don't build sentences with `+` — it breaks
  pluralisation and grammar in any future locale. Use template literals
  with named interpolations or a single key per phrase.

## Accessibility rules

`svelte-check` and the `eslint-plugin-svelte` recommended config catch
some violations. The rest is on you.

### Semantics

- **Use the right element.** `<button>` for actions, `<a>` for navigation,
  `<label>` for inputs, `<h1..h6>` for hierarchy in order, `<nav>` /
  `<main>` / `<aside>` / `<header>` / `<footer>` for landmarks.
- **No bare clickable `<div>`s.** If you absolutely need a `<div>` /
  `<span>` for visual reasons, set `role`, `tabindex="0"`, and keyboard
  handlers (`onkeydown` for Enter / Space). Prefer the right element.

### Labels

- Every form input has a programmatic label. Either:
  - A `<label for="…">` paired with `id`, or
  - `aria-label` / `aria-labelledby`.
- Icon-only buttons need `aria-label` (or visible text via an `sr-only`
  span).
- Decorative icons → `aria-hidden="true"`.

### Focus & keyboard

- Don't kill focus outlines.
- Modals / drawers / popovers — extend the existing
  [`Modal.svelte`](../../../src/lib/components/Modal.svelte) /
  [`Backdrop.svelte`](../../../src/lib/components/Backdrop.svelte)
  primitives instead of rolling your own. They handle the dimmed
  backdrop + Esc-to-close. For sign-out specifically, prefer
  [`LogoutConfirmModal.svelte`](../../../src/lib/components/LogoutConfirmModal.svelte)
  over a bespoke confirm dialog.
- Modal close button must be reachable by keyboard.
- For lists where the user picks one of N items, focus order matches
  visual order.

### Color & contrast

- Never communicate state with color alone — pair it with an icon, label,
  or shape.
- Use the semantic design tokens from
  [`src/app.css`](../../../src/app.css)'s `@theme` block via the matching
  Tailwind utilities (`bg-primary`, `text-default`, `border-border-soft`,
  `text-success`, …). Never hard-code hex, never use raw `bg-white` /
  `text-white` / `border-black` literals — they bypass the
  `[data-theme]` swap. See
  [`stack-and-patterns.md#theming--dark-mode-readiness`](./stack-and-patterns.md#theming--dark-mode-readiness).

### Images / media

- Meaningful images: `alt="…"` describing the content.
- Decorative images: `alt=""` (empty, not omitted).
- SVG icons: see [Icons](#icons) below.

### Routing

- Pandame is a multi-route SvelteKit SPA. There is one global `<title>`
  set in [`src/app.html`](../../../src/app.html); do not overwrite it
  from a child component (no `<svelte:head><title>…` per page today).
  If a route ever needs its own document title, surface the question
  first.
- `target="_blank"` requires `rel="noopener noreferrer"`.

### Icons

- Reusable icons live as their own components under
  [`$lib/components/icons/`](../../../src/lib/components/icons/) and are
  keyed by `currentColor`. If you find yourself inlining an SVG that
  appears in two components, extract it into `icons/`.
- Bespoke one-off SVGs (e.g. the close button in `Modal.svelte` or the
  dotted ring in `PandaMark.svelte`) stay inline — that's fine.
- Decorative icons inside a labelled control get `aria-hidden="true"`.
- Icon-only controls get `aria-label="…"` on the button (not on the
  icon). The shared
  [`IconButton.svelte`](../../../src/lib/components/IconButton.svelte)
  primitive enforces this via its required `ariaLabel` prop.

### ARIA — minimum surface

Use ARIA only when HTML semantics are insufficient. If you find yourself
writing `role="button"`, the answer is almost always to use a real
`<button>` instead.

## HTML safety

- Any HTML coming from backend / user input must be sanitised before
  `{@html …}`. There is no app-wide sanitiser — if you need one, surface
  the question in the PR description before adding a dependency.
- External links: `target="_blank"` requires `rel="noopener noreferrer"`.

## Quick checklist (every UI change)

- [ ] Every interactive element is a real `<button>` / `<a>` / existing
      primitive (`Button`, `Modal`, `Backdrop`, …).
- [ ] Every input has a label or `aria-label`.
- [ ] Icon-only buttons have an accessible name.
- [ ] Decorative icons / images have `aria-hidden="true"` / empty `alt=""`.
- [ ] State is conveyed by more than colour.
- [ ] No `text-#hex`, no inline `style="color:…"`, no raw `text-white` /
      `bg-white` / `border-black` literals — use the semantic tokens.
- [ ] External links: `rel="noopener noreferrer"` on `target="_blank"`.
- [ ] If untrusted HTML is rendered: sanitise first, then `{@html …}`.
- [ ] User copy goes through `$i18n.*`, not hard-coded strings.
